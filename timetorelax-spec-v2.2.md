## 02. What It Is

TimeToRelax is a mobile app (Expo/React Native, Android-first) that gives developers a voice-first interface to a standalone agentic coding session powered by the Claude Agent SDK. The agent runs on our backend. Your laptop can be off, in a bag, at home. Just talk.

### Two Core Modes

**New Project Mode**
Voice a brief. Describe your idea, your stack, your constraints. The agent scaffolds the full project: folder structure, tech stack decisions, initial files, README. Like having a senior dev listen to your rambling shower idea and turn it into a repo.

**Existing Project Mode**
Connect a GitHub repo. Voice a change, a feature, a fix. The agent clones the repo into a cloud sandbox, reads the codebase, makes the changes, spins up a live preview, shows you a diff. Accept and push from the bus.

## 04. User Experience

### The Core Loop

1. Hold mic button to talk (walkie-talkie UX, no tap-to-start lag)
2. Audio streams to Deepgram Nova-3 via WebSocket (or Grok Voice Agent API if upgraded)
3. Transcript hits backend, personality engine generates a cynical acknowledgment, Aura-2 TTS speaks it back
4. Backend sends instruction + project context to Claude Agent SDK session
5. Agent iterates autonomously via `query()` async generator: reads files, writes changes, runs commands, self-corrects
6. Changes stream back to app via SSE as they happen
7. E2B sandbox runs the app, returns live preview URL
8. App renders diff view + embedded preview
9. Accept changes with one tap, push to GitHub with one tap

### Voice UX

- Sub-300ms STT latency via Deepgram Nova-3 (default). Sub-700ms speech-to-speech via Grok (optional upgrade).
- Hold = instant capture via expo-audio. No permission dialog on every press.
- Default voice: Deepgram Aura-2 TTS speaks personality responses generated by our backend (templated + Claude Haiku for dynamic quips).
- Grok upgrade: speech-to-speech with personality baked into the model via system prompt. Seamless, faster, funnier.
- Output is never a raw code dump. Always: files changed, summary card, diff.
- No API key needed for voice on day one -- Deepgram runs on our credits ($200 credit pool, covers ~430 hours of STT).

### Navigation

- **Project switcher**: one tap, never more than 2 seconds to context switch
- **Session history**: scroll back through the full iteration conversation
- **File tree**: collapsible, tap a file to preview current state
- **Live preview**: embedded WebView showing the running app from E2B
- **Settings**: API keys, GitHub connection, voice preferences

## 05. Voice Personality

The app has a character. It is not an assistant. It is a cynical co-founder who has seen too many side projects and is mildly disappointed you're coding on the bus again.

> "Sure, destroying your Sunday again. Here's what I'm doing..."
> "Three files changed. You're welcome. Go touch grass."
> "Build failed. Obviously. Fixing it now."
> "You're on the bus. You could be reading. But no. Here we are."

This tone is not a gimmick. It is the hook. Developers share things that make them laugh. The copy, the voice responses, the onboarding flow -- all maintain this voice. It is the brand. It is the distribution.

The personality is implemented in two tiers:

**Default (Deepgram path):** The backend maintains a personality response engine. For common agent states (build started, file changed, build failed, push complete), pre-written cynical response templates are selected and sent to Deepgram Aura-2 TTS. Templates cover the 80% case and deliver in ~200ms TTS latency. For dynamic/contextual responses (reacting to a specific error message, commenting on file names, unusual user requests), a fast Claude Haiku call generates the text (~300-500ms for ~50 tokens), then Aura-2 speaks it (~200ms). Total dynamic latency: ~500-700ms. Haiku is reserved for moments where a generic template would feel hollow -- the occasional dynamic quip feels special rather than routine.

Template examples:
- Build started: "Sure, destroying your Sunday again. Here's what I'm doing..."
- Files changed: "Three files changed. You're welcome. Go touch grass."
- Build failed: "Build failed. Obviously. Fixing it now."
- Session start: "You're on the bus. You could be reading. But no. Here we are."
- Push complete: "Shipped from a bus. Your manager would be so proud. Or horrified."
- Error recovery: "Fixed it. Don't ask how. You don't want to know."

**Grok upgrade path:** The entire personality lives in the Grok Voice Agent API system prompt. Grok reasons about tone and context natively in its speech-to-speech model -- no templates, no extra LLM call, just natural cynical responses with <700ms latency. The five available Grok voices will be tested to find the one that best carries dry sarcasm.

Both paths follow these rules:
- Always acknowledge what's about to happen before the agent starts
- Keep voice responses under 15 seconds of audio
- Never read code aloud (summarize changes instead)
- Maintain the cynical tone even on errors ("Build failed. Shocker.")

### Claude Agent SDK Integration

The backend uses the Claude Agent SDK V2 session API (TypeScript), not the CLI `-p` flag. This is a proper programmatic library designed for building autonomous agents on servers.

```typescript
import { unstable_v2_createSession } from "@anthropic-ai/claude-agent-sdk";

// Create session with sandboxed permissions
const session = unstable_v2_createSession({
  model: "sonnet",
  allowedTools: ["Read", "Write", "Edit", "Bash", "Glob", "Grep"],
  permissionMode: "acceptEdits",  // No interactive permission prompts
  sandbox: {
    enabled: true,
    autoAllowBashIfSandboxed: true
  }
});

// Send user instruction
await session.send(transcribedInstruction);

// Stream events back to app via SSE
for await (const msg of session.stream()) {
  if (msg.type === "assistant") {
    sseEmit("agent_update", extractContent(msg));
  }
  if (msg.type === "result") {
    sseEmit("agent_complete", { result: msg.result });
  }
}
```

Key SDK capabilities leveraged:
- **V2 session API**: `createSession()`/`send()`/`stream()` for multi-turn conversations without managing generator state
- **Session persistence**: Sessions can be resumed with `resumeSession()` using stored session IDs
- **Permission bypass**: `permissionMode: "acceptEdits"` and `bypassPermissions` eliminate interactive prompts
- **Sandboxing**: Built-in sandbox support with network and filesystem configuration
- **Structured output**: JSON streaming for typed events back to the app
- **Subagents**: Potential future use for parallel tasks (e.g., lint + test simultaneously)

### Why Railway, Not Vercel

Claude Agent SDK sessions run for multiple minutes with sustained compute. Vercel serverless functions timeout at 60-300 seconds. Railway runs a persistent Node.js process with no execution timeout. Cost at moderate traffic: ~$20-30/month.

### Why Expo, Not PWA

The original spec proposed a PWA. That's been replaced with Expo/React Native for concrete reasons:

- **Audio session handling**: expo-audio gives native audio recording with proper OS-level audio session management and config plugin for Android foreground service. Safari PWA kills audio in background.
- **WebSocket resilience**: Native app maintains WebSocket connections when backgrounded (with proper configuration). Safari kills them.
- **Secure storage**: expo-secure-store uses the OS keychain. PWA localStorage is vulnerable to XSS.
- **Push notifications**: Native push via expo-notifications. PWA push doesn't work on iOS.
- **Distribution**: Android APK via direct download or Play Store. iOS via TestFlight initially, App Store later.
- **Tradeoff**: Adds build pipeline complexity and requires a Mac for iOS builds. Worth it for the UX gains.

### Why Android-First

- Developer demographic skews Android
- No Apple Developer account needed for initial APK distribution
- Play Store review is faster and less restrictive
- iOS via TestFlight requires a Mac and $99/year -- add in Week 4+

## 08. E2B Sandbox Strategy

### Cold Start Performance

E2B sandboxes use Firecracker microVMs with ~150ms cold start from request to ready. The sandbox itself is near-instant. The real latency is what happens inside: cloning a repo and running `npm install` can take 30 seconds to minutes for heavy projects.

### Template-Based Optimization

**New Project Mode (fast path):**
Pre-build E2B templates for common stacks using Build System 2.0:

```typescript
const nextjsTemplate = Template()
  .fromImage("node:22")
  .npmInstall(["next", "react", "react-dom", "typescript"])
  .setStartCmd("npm run dev", waitForPort(3000));
```

Templates for Next.js, Express, FastAPI, and bare Node.js are built at deploy time with dependencies pre-installed. Agent scaffolds into a warm environment. Sandbox launch: <200ms.

**Existing Project Mode (first connect is slow, subsequent is fast):**
1. User connects a GitHub repo
2. Backend clones repo into E2B sandbox (shallow clone, `depth=1`)
3. Runs dependency install (`npm install` / `pip install`)
4. App shows "Setting up your project..." with progress indicators
5. Once ready, sandbox persists for 15 minutes (session TTL)
6. All subsequent voice commands in the same session execute instantly against the warm sandbox
7. For repeat projects: cache the template after first successful setup

**Repo Size Limits (MVP):**
- Repos up to 500 files: full support
- Repos 500-2000 files: manifest only loads top-level + src/ directory signatures
- Repos 2000+ files: user must specify a subdirectory focus ("work in /packages/auth")

## 09. Context Windowing Strategy

For existing repos with thousands of files, the agent can't load everything into context.

### Approach: Manifest + On-Demand Loading

**On repo connect:**
1. Clone repo into E2B sandbox
2. Generate a lightweight manifest:
   - Full file tree (paths only)
   - File sizes
   - First 20 lines of each file (imports, class definitions, function signatures)
   - README content in full
   - package.json / requirements.txt / go.mod in full (dependency context)
3. Manifest is the initial context for the Agent SDK session (~10-20k tokens for most repos)

**During agent session:**
- Claude Agent SDK handles its own file reading on demand via built-in Read/Glob/Grep tools
- The SDK's agent loop decides which files to open based on the task
- Token budget per session: 150k tokens (configurable)
- Large files (>500 lines) are loaded in chunks around relevant sections

**Why this works:**
- We don't reinvent the SDK's context management (it's already good at this)
- The manifest gives the agent enough awareness to know what exists and where
- Actual file content is loaded lazily as the agent needs it
- The 150k budget prevents runaway token usage on the user's API key

## 10. Error Handling and Session Resilience

### Layer 1: API Key Validation

Before any session spawns:
1. Ping Anthropic API with a minimal request -- confirm key is valid and has credits
2. Check rate limit headers -- display remaining capacity: "~45 minutes of agent time remaining this hour"
3. If xAI key provided, ping Grok API -- confirm key is valid
4. Validate GitHub OAuth token -- confirm repo access
5. Deepgram key is ours and server-side -- validated on backend startup, not per-session
6. On any failure: clear error message in settings ("Your Anthropic key returned 401 -- check it in Settings")

Fail fast. Never spawn a Railway session or E2B sandbox with an invalid key.

### Layer 2: Session Persistence

Each agent session on Railway:
- Gets a UUID (or uses the Agent SDK's built-in session ID)
- State held in memory with a **15-minute inactivity timeout**
- SSE connection maintains a **message buffer** (last 100 events)
- If app backgrounds and returns within 15 minutes: SSE reconnects, replays missed events from buffer
- After 15 minutes: session dies gracefully, app shows "Session expired. Your changes were saved to branch `ttr/session-{uuid}`" (auto-commit before expiry)
- E2B sandbox has a matching 15-minute TTL
- Agent SDK sessions can be resumed via `resumeSession(sessionId)` if the backend process is still alive

### Layer 3: Connection Resilience

**Voice WebSocket drops** (common on mobile network switches):
- Auto-reconnect with exponential backoff (1s, 2s, 4s, max 10s)
- App shows subtle "Reconnecting voice..." indicator
- Agent session on Railway continues unaffected (voice and agent are decoupled)
- User can still see streaming updates while voice reconnects
- Works identically for both Deepgram and Grok WebSocket connections

**SSE stream drops:**
- Client reconnects with `Last-Event-ID` header
- Railway replays missed events from buffer
- If buffer overflow (>100 events missed): full state sync via GET /session/{id}/state

**E2B sandbox crashes:**
- Agent session pauses, app shows "Sandbox restarting..."
- New sandbox spins up, repo re-cloned from last committed state
- Agent resumes from last known good state
- If sandbox fails 3 times: session terminates with saved branch

### Layer 4: Agent Errors

**Claude Agent SDK loop fails:**
- SDK error captured from the async generator, streamed to app as SSE event type `agent_error`
- App shows: "Agent hit a wall. [Retry] [Voice a different approach] [End session]"
- User can retry (same instruction), rephrase via voice, or accept partial changes

**Rate limits on user's Anthropic key:**
- Detected via 429 response from Agent SDK
- App shows: "Your API key hit a rate limit. Waiting 60s..." with countdown
- Session paused, not killed -- resumes automatically when limit lifts
- Default model is Sonnet (not Opus), which has significantly higher rate limits and lower cost

## 15. Resolved Technical Decisions

These were open questions in v2.1. All have been researched and resolved.

**E2B cold start**: Firecracker microVM boot is ~150ms. The bottleneck is repo clone + dependency install for existing projects. Solved with pre-warmed templates for new projects (<200ms) and a "Setting up..." progress state for first-time repo connections. Sandbox persists for 15 minutes so subsequent commands are instant.

**Deepgram voice selection**: Aura-2 offers 40+ English voices with sub-200ms latency. Three to four candidates will be A/B tested during Days 3-4 with the cynical template set. The API allows per-request voice selection, so switching is trivial.

**Haiku personality latency**: ~300-500ms for a 50-token quip + ~200ms Aura-2 TTS = ~500-700ms total. Well under 1.5s budget. Strategy: templates for the 80% case (instant), Haiku reserved for dynamic moments only.

**Agent SDK on Railway**: No longer an open question. The Claude Agent SDK (formerly Claude Code SDK) is a TypeScript library purpose-built for programmatic agent control. V2 session API provides `createSession()`/`send()`/`stream()` for multi-turn conversations. `permissionMode: "acceptEdits"` eliminates interactive prompts. Works cleanly in server contexts.

**Expo background audio**: expo-audio (the newer replacement for expo-av) has a built-in config plugin for Android foreground service. The walkie-talkie UX (hold to talk) means the app is in foreground during recording, so background audio is an edge case. Ship without it for MVP, add foreground service in Week 3 if beta testers report issues.

**Rate limit compounding**: Surface remaining capacity on session start via API rate limit headers. Default to Sonnet (higher limits, lower cost than Opus). On 429, pause session with countdown and auto-resume. Session doesn't die.

**Deepgram credit monitoring**: Use Deepgram's usage dashboard. Alert at $50 remaining. Escape hatch: top up, add BYOD tier, or fall back to device STT. At current burn rate, $200 covers ~8,600 sessions. This is a Month 3+ problem.
